## 发现 1: CC-Bank AdaIN 公式历史 hybrid bug (research-pipeline → 代码 review 阶段)

**时间**: 2026-04-22 23:20

**背景**: EXP-119 Sanity B 实现 `fedbn_ccbank.py` 时，参考 MixStyle/FedCA 写 AdaIN 实现。Self-review 发现代码里有一个**语义不匹配**的数学错误。

**具体问题**:
- 原 `_compute_aug_loss` 里的 self-normalization 统计量:
  ```python
  mu_self = h.mean(dim=1, keepdim=True)   # [B, 1]  per-sample scalar
  sigma_self = h.std(dim=1, keepdim=True) + 1e-5  # [B, 1]
  ```
- 但 bank 里存的 style 参数 `mu_other, sig_other` 是 **per-feature [D]** (通过 `_compute_class_stats` 对 class c 的 batch 做 per-feature 均值/方差)
- 实际相当于 "scalar LayerNorm + per-feature style injection" 的 hybrid, 语义不统一

**数学后果**: 注入的 class-c style 特征会因为 scalar normalization 过度压缩 sample 自己的方向。可能是历史上 EXP-040/059/061/047 四次 AdaIN 失败的一个未知原因。

**修复** (commit 2ed462e):
```python
mu_self = h.mean(dim=0, keepdim=True)                         # [1, D]
sigma_self = h.std(dim=0, keepdim=True, unbiased=False) + 1e-5  # [1, D]
h_normalized = (h - mu_self) / sigma_self                     # [B, D]
# AdaIN: σ_other * (h - μ_self_batch)/σ_self_batch + μ_other
h_aug[i] = h_normalized[i] * sig_other + mu_other             # [D]
```

**How to apply**: 任何在 pooled feature (非 conv) 上做 AdaIN 的代码，self-normalization 必须与 style injection 同维度（都 per-feature 或都 scalar）。pooled feature 没有 spatial 可 average，标准 MixStyle per-sample-per-channel 语义不适用。

## 发现 2: 单元测试是 shadow test，没严格测算法代码

**时间**: 2026-04-22 23:25

**背景**: 发现 1 修复 AdaIN 公式后，跑 18 个 CC-Bank pytest 单测，**全部 pass**。但仔细看测试代码：

```python
# test_adain_formula_consistency
mu_self = h.mean(dim=1, keepdim=True)      # 测试自己写的公式
sigma_self = h.std(dim=1, keepdim=True) + 1e-5
...
h_adain = ((h[0] - mu_self[0]) / sigma_self[0]) * sig_other + mu_other  # 测试自己算
# 断言 h_adain.mean() ≈ mu_other.mean()
```

**问题**: 测试代码**自己复现 AdaIN 公式**做断言, 而非 `from algorithm.fedbn_ccbank import Client; ...; client._compute_aug_loss(...)` 直接调算法代码。

**后果**:
- 算法代码改了, 测试没改 → 测试仍过, 但其实在测 "测试自己写的公式"
- 这种"影子测试"只验证数学语义, 不验证代码实现
- 相当于 38 个单测里至少 3-5 个是 shadow test

**How to apply**: 未来写算法单测, 必须直接 `import` 算法类并调用方法; 禁止自己在测试文件里重写算法公式。可以写辅助 fn 比对 reference 实现, 但断言应该是在算法返回值上。

## 发现 3: RTX 5090 (Blackwell sm_120) 与 torch 2.1.2+cu118 完全不兼容

**时间**: 2026-04-22 23:05

**背景**: 新 AutoDL 实例给配了 RTX 5090 32GB, 期望 1.68 元/h 跑 Sanity runs。

**测试**:
```python
x = torch.randn(100, 100).cuda()
y = x @ x
# RuntimeError: CUDA error: no kernel image is available for execution on the device
# RTX 5090 with CUDA capability sm_120 is not compatible
# Current PyTorch supports sm_50..sm_90
```

**根因**: torch 2.1.2 编译时只 target 到 sm_90 (H100). RTX 5090 是 sm_120 (Blackwell)，需要 **torch 2.7+cu128** 才支持。

**解决路径**:
- 升级 torch (`pip install torch==2.7 --index-url .../cu128`) ~2-3GB 下载
- 或换卡 (RTX 4090 sm_89, A100 sm_80)

**实际选择**: 用户直接开了新 RTX 4090 24GB 实例 (westb:13399), 免去升级。

**How to apply**: AutoDL 选卡时，凡是**实例描述里有 "5090" / "RTX 50 系" / "Blackwell"** 都要先 check torch 版本是否支持 sm_120, 否则锁 RTX 4090 / A100。sm 对照: RTX 30 = sm_86, RTX 40 = sm_89, A100 = sm_80, H100 = sm_90, RTX 50 = sm_120。

## 发现 4: flgo benchmark Model 把 AlexNet 包在 self.model (FModule wrapper)

**时间**: 2026-04-22 早期 (EXP-119 开发时)

**背景**: 写 `fedbn_ccbank.py` 的 `forward_with_feature(model, x)` 调用 `model.features(x)` 时报 `AttributeError: 'Model' object has no attribute 'features'`。

**根因**: `benchmark/{office,pacs,domainnet}_classification/model/default_model.py` 里:
```python
class Model(FModule):
    def __init__(self, ...):
        self.model = AlexNet(...)
    def forward(self, x):
        return self.model(x)
```
FModule 是 flgo 的 wrapper, AlexNet 被包在 `self.model`。我们写算法调 `.features` 会找不到。

**修复**: 加 `_unwrap_alexnet(model)` helper, 走 `.model` / `.net` 链找到真的 AlexNet:
```python
def _unwrap_alexnet(model):
    inner = model
    for _ in range(3):
        if hasattr(inner, 'features'):
            return inner
        if hasattr(inner, 'model'):
            inner = inner.model; continue
        ...
```

**How to apply**: 任何自定义算法想访问 AlexNet 内部层 (`.features`, `.fc1`, `.bn6` 等) 都必须先 unwrap。写算法时统一用这个 helper, 不要假设 `model` 直接是 AlexNet。

## 发现 5: 在 algorithm 文件里定义空 init_global_module = silent kill model construction

**时间**: 2026-04-22 早期

**背景**: 写新 algorithm 文件时, 最后加了:
```python
def init_global_module(server, client_data):
    pass
def init_local_module(server, client_data):
    pass
def init_dataset(server, client_data):
    pass
```
以为这是 flgo 约定的 hook。结果跑起来 `AttributeError: 'NoneType' object has no attribute 'get_device'` 在 Initial Evaluation。

**根因**: flgo 发现 algorithm 有这些函数就不再去 benchmark 里找 default 的 model 构造 hook。**空 pass 直接让 server.model = None**。

**修复**: 删掉这 3 个空函数。让 flgo fallback 到 benchmark 的 default_model.py。

**How to apply**: 写新 algorithm 文件时, 如果不需要定制 model/data 构造, **完全不要定义 init_global_module/init_local_module/init_dataset**。需要定制才定义。空 pass 是最坏的选择（比没写还糟）。

## 发现 6: 🔴 正交头对 Best accuracy **贡献几乎为零** (EXP-116 颠覆性结果)

**时间**: 2026-04-22 晚 (EXP-116 6/6 run 全部完成回填)

**背景**: EXP-080 orth_only (lo=1.0) PACS 80.41 / Office 89.44 接近或超 FDSE. 过去一直认为 "正交损失 cos²(z_sem, z_sty)" 是 FedDSA 的核心创新。EXP-116 做干净对照: 仅改 `lo` 从 1.0 → 0.0, 其他完全不变.

**数据**:

| 指标 | lo=1 (EXP-080) | lo=0 (EXP-116) | Δ |
|---|:---:|:---:|:---:|
| PACS AVG Best | 80.41 | **80.47** | **+0.06** (lo=0 反而略高!) |
| PACS AVG Last | 79.42 | 78.91 | −0.51 |
| Office AVG Best | 89.44 | **89.75** | **+0.31** (lo=0 反而略高!) |
| Office AVG Last | 88.71 | 86.89 | −1.82 |

**关键观察**:
1. **Best 差全部 < 0.3pp, 而且方向是 lo=0 更好** — 正交损失对 Best accuracy **零贡献**甚至轻微伤害
2. **Last 差为负** (lo=1 比 lo=0 Last 好 0.5-1.82pp) — 正交损失只做了轻微 **后期稳定化**, 防止训练末段 z_sem 退化
3. **不满足** "lo=0 < FDSE (79.91 / 90.58)" 条件 — lo=0 PACS 80.47 仍胜 FDSE +0.56

**对 paper 叙事的直接颠覆**:
- ❌ 不能再卖 "正交头几何解耦 z_sem / z_sty 是核心创新"
- ✅ 真正的 accuracy 来源: **SGPA 双头架构本身** (fc1→bn6→fc2→bn7 两个 256d 头的存在) + **LR=0.05 + lr_decay 0.9998 优化 schedule**
- ✅ 正交损失 = 可选的轻微 regularizer, **不是**性能核心

**How to apply**:
1. EXP-119 FedPTR Sanity 方案里不要再把正交头作为必含组件卖点
2. EXP-117 DomainNet orth_only 数据回来后, 如果也 beat FDSE, 进一步证明 **架构 > 损失**
3. Paper Method 章节的"Motivation"和"Loss Design"部分需改写, 避免 reviewer 问 "你们的 ablation 显示正交头无 Best 贡献, 凭什么说它是 key contribution?"
4. 下一轮 research-pipeline L4 (experiment-plan) 必须补 "纯 SGPA 架构 without orth / without 任何辅助 loss" 的裸跑对照 baseline

## 发现 8: 完整 FedBN vs 半 FedBN Office 无差异 (EXP-118)

**时间**: 2026-04-22 12:26 (EXP-118 Office 3/3 完成)

**背景**: feddsa_scheduled 一直用 "半 FedBN" 实现 (γ/β 参与 FedAvg, 只 running stats 本地), 没做过"完整 FedBN"对照. 潜在担忧: γ/β 聚合可能是个未被注意到的性能漏洞.

**数据** (Office AVG Best, 3-seed mean):

| 方法 | Office AVG Best |
|---|:---:|
| FDSE (baseline) | 90.58 |
| **EXP-080 半 FedBN** (γ/β 聚合) | 89.44 |
| **EXP-116 半 FedBN lo=0** (γ/β 聚合, 无正交) | 89.75 |
| **EXP-118 完整 FedBN** (γ/β 本地) | **89.68** (Δ=+0.24 vs EXP-080) |

**关键观察**:
1. `|Δ Best (fullbn - halfbn)| = 0.24 < 0.3pp` — **差异无实质**, 属于 seed 方差范围内 (std ~1.5pp)
2. Last 差 −0.13pp 也无实质
3. EXP-116 (89.75) 和 EXP-118 (89.68) 两者差 0.07pp — 再次验证 BN 设计细节不贡献 accuracy

**对 paper 叙事影响**:
- ✅ 半 FedBN (EXP-080 接口) 是 safe 的, 不需要重写代码
- ✅ Reviewer 问 "你们是完整 FedBN 还是半 FedBN? γ/β 本地化吗?" — 回答 "半 FedBN 对齐 ICLR 2021 原版, 完整 FedBN 测试只带 +0.24 微弱增益在 seed 方差内"
- ⚠️ 配合**发现 6** (正交头无 Best 贡献), 进一步证明: **BN 设计 + 正交损失都不是性能来源**. 真正 accuracy 来源是 SGPA 双头架构 + 优化 schedule.

**执行坑 (记录以防重踩)**:

1. **flgo init_global_module 依赖**: 从别的算法继承时必须 `from algorithm.feddsa_scheduled import init_global_module, init_local_module, init_dataset`. 否则 flgo fallback 到 benchmark default_model 没 `encode` 方法, 立刻崩.

2. **JSON 文件名超长 (Errno 36)**: feddsa_scheduled_fullbn + 13 个 algo_para + 标准超参后缀 拼成的 record filename = 260+ 字符, 超 Linux 255 字节文件名上限. **所有 3 seed 跑完 R200 但 JSON 全部没保存**, 数据只在 log 文件里. 下次写衍生算法要么: (a) 算法名简短 (fullbn 已经算短了, 但 algo_para 太多), (b) override flgo logger 截断 filename, 或 (c) 手动 save_output_as_json 到短文件名.

**How to apply**:
1. 写衍生算法 (A_plus_B 形式) 时, 必须 `from base_algo import init_*` 避免 silent model fallback
2. 监控 log 结尾是否有 "File name too long" 错误 — 如果有, 数据只在 log 里, 必须 grep log 提取
3. Paper 方法描述一句话: "We use half-FedBN (γ/β aggregated, running stats local) following FedBN [ICLR 21]. Switching to full-FedBN (all BN params local) changes Office accuracy by <0.3pp, well within seed variance (Sec. X.Y)."

## 发现 9: 🔴 "Centralized 是 method-space 上限" 在多域数据上**伪命题** (EXP-119 Wave 1)

**时间**: 2026-04-23 00:35 (EXP-119 Wave 1 完成)

**背景**: EXP-119 Sanity A (C1) 设想 centralized AlexNet 跑 Office 能到 ≥ 93 作为 method space 上限判断. 实际跑出来 **Office AVG Best = 81.81** 远低于 FDSE 90.58, 甚至低于 FedBN 88.68.

**数据**:

| 方法 | Office AVG Best | 备注 |
|---|:---:|---|
| FDSE | 90.58 | per-client DSE 本地 + BN 本地 |
| FedBN (旧 baseline) | 88.68 | BN 本地 (running stats) |
| EXP-116 lo=0 | 89.75 | FedBN + 无正交 |
| **Centralized Office** | **81.81** | **global BN + 合并训练, 4 domain 混合** |

**根因分析**: centralized 训练时 batch 是 4 个 domain 混合的, BN running_mean/var 学到的是"4 domain 混合均值". test 时 per-domain 评估 global BN stats 不匹配单个 domain 真实分布 → per-domain acc 天然降低. 不是 AlexNet capacity 问题, 是 **BN 在多域上的固有行为**.

**证据**: 
- Centralized 81.81 < FedBN 88.68 → "BN 本地化" 带来 +6.87pp
- 远超 method space 改进上限 (+1-2pp)
- Centralized 不是 FL 上限, 是多域场景下的**下限**

**对 C1 判决的影响**: 
- 原实验设计假设 "centralized > 93 = method space 有空间" 是**错的**
- 正确的 capacity test 应该是 **per-domain single training** (4 domain 各训一个模型) 或 **centralized + per-client BN** (等价于 FedBN, 已知 88.68)
- 不能因为 centralized 81.81 就否决 backbone 或换 ResNet-18

**How to apply**:
1. 以后设计 capacity test **必须考虑 BN 对多域数据的处理**, centralized + global BN 在 cross-domain 上有结构性天花板
2. 真正的 method space 上限 = **per-domain single training** 的 acc 平均 (这才是"每 client 自己训自己"的上限)
3. Paper 写作时**不要**把 "centralized baseline" 写成 "upper bound" — 要写 "global BN baseline", 说明它是受限的. 真正的 oracle 是 domain-specific training
4. Paper 可以做个 figure: {centralized global BN / centralized local BN / FedAvg / FedBN / FDSE / Ours} 展示 BN 本地化贡献远大于 loss 设计贡献

## 发现 10: CC-Bank α=0.5 Office 89.35 — 没打过 FedBN baseline (AdaIN 第 5 次"部分失败")

**时间**: 2026-04-23 00:35 (EXP-119 Wave 1 完成, Wave 2 α=0.3/0.7 还没完)

**背景**: EXP-119 Sanity B (C2) 验证 class-conditional feature AdaIN (CC-Bank) 是否帮助. 历史上 EXP-040/059/061/047 四次 AdaIN 都挂 (PACS -2.54%, Office 崩).

**数据** (Office 3-seed mean AVG Best):

| 方法 | Office AVG Best |
|---|:---:|
| FedBN 旧 baseline (paper 引用) | 88.68 |
| **EXP-116 lo=0 (我们最强 FedBN)** | **89.75** |
| EXP-080 半 FedBN lo=1 | 89.44 |
| EXP-118 完整 FedBN | 89.68 |
| **EXP-119 CC-Bank α=0.5** | **89.35** |

**判决**:
- **技术过线 (> 89.18 = 88.68 + 0.5)** ✅ — 按 NOTE 原阈值
- **没过新 baseline (< 89.75)** ❌ — 我们自己的 FedBN 变体已经更高
- CC-Bank 是历史 AdaIN 第**五次部分失败** — 没崩 (不像 EXP-059 -2.54%), 但也没赚到

**为什么没崩**: 
1. 我们做了 batch-per-feature AdaIN (发现 1 修的 bug), 不像原版 scalar-hybrid
2. AdaIN 位置在 pooled feature (1024d), 不在深层 conv (MixStyle 研究说深层 AdaIN -7%)
3. Class-conditional 筛选避免了 "某类样本少 μ/σ 估计误差"

**为什么没赚**: 
- Office 4 domain 都是办公物品, 风格异质弱 (见发现 7)
- FedBN 的 BN 本地化已经处理掉大部分跨域差异
- AdaIN 额外加的 "cross-client style augmentation" 在弱风格异质场景没信号

**不立即 kill CC-Bank 的理由**:
1. **Wave 2 α=0.3 和 α=0.7** 还没完 — α 可能要调
2. **PACS 是真正的风格异质场景** — Wave 3 PACS 结果决定 CC-Bank 命运
3. **AdaIN 在弱异质没效果** 是已知现象, 不代表它在强异质也没效果

**How to apply** (给未来决策):
1. Wave 3 PACS 完成后, 如果 CC-Bank PACS AVG Best > 80.41 (EXP-080 lo=1) + 0.5 → 保留, paper 主卖 "CC-Bank 对强异质有效"
2. 如果 Wave 3 PACS CC-Bank < 80.41 + 0.5 → **正式 kill CC-Bank**, 转向其他路线
3. 不能因为 Office 单一数据集结果就决定 CC-Bank 生死 — 它本来就是为风格异质设计的

## 发现 12: 🔴 chained wave dispatcher 浪费 GPU 半天 (EXP-119 部署 lesson)

**时间**: 2026-04-23 01:30 (发现 + fix)

**背景**: EXP-119 原 dispatcher 设计:
```
Wave 1 Office 6 runs (1h) → wait → Wave 2 Office 6 runs (1h) → wait → Wave 3a PACS 4 runs (7h) → wait → Wave 3b (7h) → wait → Wave 3c (7h)
Total: ~23h wall
```

**实际**: Wave 2 Office 6 runs 只用 13GB/24GB 显存, 剩 11GB 完全空. **同期完全可以并行 2-3 个 PACS runs**, 但 dispatcher 的 `wait` 让 GPU idle 等待.

**Fix**:
1. Kill 串行 dispatcher, 保留已 launched python 继续跑
2. 写 `greedy_pacs.sh` — 按 `nvidia-smi --query-gpu=memory.free` 动态 launch, 阈值 4500MB (每个 PACS run 的显存预估)
3. 每 20s check 一次 free memory, 有空位立即 launch 下一个 task, 没空位 sleep 45s 再 check

**效果**: 
- 原: 3 批 × 4 并行 × 7h = 21h PACS wall
- 新: 12 并行 × 7h = 7h PACS wall (显存允许, 实测 GPU 20.3/24GB, 53% util)
- **节省 14h wall time ≈ 半天**

**规则写入 CLAUDE.md 17.8 GPU 并行原则 (强制)**:
- ❌ 禁止 chained wait dispatcher
- ✅ 用 greedy scheduler 按显存动态 launch
- ✅ Launch 后 `sleep 15-20s` 让进程 ramp up 再 check 下一个

**配套 JSON 覆盖坑 + watcher 救场**:

另外发现 flgo `save_output_as_json` 的 record filename **不含 algo_para**, 所以同 algorithm 不同 config (如 CC-Bank α=0.3/0.5/0.7) 的 JSON 会**互相覆盖**. 
- EXP-119 Wave 1 α=0.5 JSON 被 Wave 2 α=0.3 覆盖 (幸好 Wave 1 数据已提取过)
- Wave 3 trajectory η=0/0.5 也会互相覆盖

**Fix**: 写 `json_watcher.py` 每 30s 扫 record/, 检测到 JSON mtime 变化就立即 cp 带时间戳 backup (`*_T<timestamp>.json.bak`). 这样即使正 JSON 被覆盖, backup 保留所有版本.

**How to apply** (未来实验):
1. 用 greedy scheduler 不用 chained dispatcher
2. 每次跑 α/η scan 时, 必须 check algorithm 的 record filename 是否含 algo_para, 不含就启动 watcher
3. 或者更根本: 改 flgo save_output_as_json 让 filename 自动含 algo_para 后缀
4. 更更根本: 每个 variant 跑在独立 task dir 下 (但 IO 开销大, 不推荐)
