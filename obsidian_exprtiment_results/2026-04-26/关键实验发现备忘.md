## 发现 8: F2DC release 代码 8 个 bug 才能跑通 (2026-04-26)

**时间**: 2026-04-26 21:30-22:00

**事件**: 复现 F2DC (CVPR 2026 录用论文) 时, GitHub release 代码在 PyTorch 2.6 + Python 3.10 上**直接踩 8 个 bug**:
1. `backbone/VGGNet_DC.py` 完全缺失 (datasets/__init__.py 全量 import 触发崩)
2. `backbone/WRN.py` 完全缺失
3. `datasets/{pacs,officecaltech,digits}.py` 都用 `selected_domain_list == []` 比较 numpy array, shape mismatch 崩
4. `datasets/utils/federated_dataset.py:218,353` partition 算法 `np.random.randint(len(prob_del_place))` 在 prob_del 空时炸
5. `backbone/ResNet_DC.py:162` forward 返回 5-tuple 但 `f2dc.py:_train_net` 期望 7-tuple
6. `models/f2dc.py:94` `DFD_sep_loss = torch.tensor(0.); DFD_sep_loss += [B]` PyTorch 2.x 不再兼容 (scalar broadcast to vector)
7. `models/f2dc.py:_train_net` rand_dataset 把同 domain 给 >3 client 时 (30% × N > 100%) 后面 client 空 dataloader 触发除零
8. `utils/training.py:29` eval 也按 5-tuple 解 forward

**根因分析**: CVPR 录用 ≠ 代码即用. release 代码很可能是早期分支或匆忙开源版本, 论文报数 (PACS 76.47 / Office 66.82 / Digits 87.23) 用的是更完整的内部代码.

**含义 (paper 角度)**:
- 我们复现 F2DC 拿到的数字才是公平 baseline
- 如果差距 > 3pp, **paper 主表只用我们复现的数字**, 不用论文报数
- 这个发现也是写 paper 时一个值得提的诚信点 (footnote: "我们用 F2DC 官方 release patches 后跑出 X%; 论文报 Y%, 差距源于 release 代码不完整")

**含义 (策略角度)**:
- F2DC 在 PACS/Office 上**很可能没我们想象的那么强**
- 我们 FedDSA orth_only 已经在 PACS 上超过 FDSE 79.91 → 80.64 (+0.73), F2DC 论文报 PACS 76.47 — 但**这是论文数字不是我们环境**, 等 EXP-130 复现完才能横向对比
- 如果 F2DC 复现后 PACS 只到 73-75 (论文 -3pp), 我们 FedDSA 80.64 直接领先

**修复证据**: git commit `25ab52a fix(F2DC): 6 个 patch 让 release 代码在 PyTorch 2.6 上跑通` (实际 8 个 bug, 提交 message 当时只数到 6 个, 启动 EXP-130 smoke 时又找到 2 个 — digits.py / officecaltech.py 同样的 array 比较 bug). 全部 patch 入 git, 后续 sc 重启 / 换实例 git pull 即可.

**实战影响**: F2DC × PACS R=5 烟雾测试 PASS (acc 10.5→30.9 单调上涨), F2DC × Office R=3 PASS (R=2 acc 19.37). 9 个 smoke (3 algo × 3 dataset) 跑完后启 EXP-130 27 runs 正式实验.


## 发现 9: PG-DFC 方案设计完成 — 给 DFC 加 prototype 引导 (2026-04-26)

**时间**: 2026-04-26 23:00 设计敲定

**事件**: 经过 8 轮迭代讨论 (PG-DFC v1 → MC-FL → EU-DFC → 双 head + proto → 最终 PG-DFC v6), 敲定基于 F2DC 的小改动方案. 详细设计文档在同目录 `PG-DFC方案_Prototype引导的F2DC校准.md`.

**核心改动**:
- F2DC 原版 DFC 是"盲修复" — 只看本地 nr_feat, 没外部参考, 靠 CE 反向梯度盲目摸索
- PG-DFC: 给 DFC 输入加上 server 跨 client 聚合的 class prototype bank, 通过 attention 让 DFC 知道往哪个 class 方向救
- 改动量约 200 行, 完全兼容 F2DC (proto_weight=0 时退化成原版)

**关键设计决策**:
- prototype 用 r_feat (robust feature) 提取, 不用 raw feat (避免 domain 污染)
- channel-level (C-dim vector), 不用 spatial (通信省 100×)
- attention 用 dot-product + /√C scaling (不用 cosine, 训练稳)
- EMA α=0.99 + warmup 30 round (前期 prototype 是噪声, 不能直接用)
- server 聚合按 class count 加权 (n_c_k / Σ n_c_k), 缺类的 client 自动 mask

**Novelty 论证 (跟现有工作差异)**:
- FedProto / FPL: 用 prototype 但没在 F2DC decouple 框架下
- Disentanglement-Then-Reconstruction (2020): 用 prototype reconstruction 但是 UDA 不是 FL
- RFedDis: 双 head + evidential 但没 prototype
- F2DC: 有 DFC 但没用 prototype 引导
- **首次在 F2DC 框架下用跨 client prototype + attention 引导 DFC 校准** = 我们的 contribution

**预期增益**: PACS 76.47 → 78+ (+1.5pp), Office 66.82 → 68+ (+1.5pp). 预期不是承诺, 实际 -0.5 ~ +2pp 之间, 必须实测.

**4 个增益来源拆解**:
1. 消除 F2DC DFD/DFC 梯度冲突: +0.3-0.5pp
2. Prototype 正向 pull 比 wrong-high 反向 push 稳: +0.3-0.5pp
3. 跨 client prototype 信号 (F2DC/RFedDis 都没有): +0.5-0.8pp (本质 FedProto 增益)
4. 兼容性 + warmup 设计避免破坏 F2DC: 不退步

**风险**:
- attention 权重塌缩 (全集中在 1-2 类) → 加 temperature τ=0.5
- prototype EMA 早期不稳 → warmup 30 round
- proto_weight 调参敏感 → grid search {0.1, 0.3, 0.5, 0.7}

**时间表**: 等 F2DC sanity (EXP-130) 复现完成 → W2 立刻实施 PG-DFC v1.

**衔接已有工作**:
- 不是从 0 开始, 复用我们 FedDSA 的 sem_head/sty_head 双头训练经验
- EXP-129 functional 诊断 (DIAD/CCTM) 直接当 evaluation 工具
- EXP-128 DualEnc 失败经验写入 paper "Failed Attempts" 章节

**重要教训** (8 轮 critique 演化):
1. 不能凭印象引数据 — 必须查 paper Tab (二版犯过)
2. 不能凭 abstract 推方法 — 必须读源码 (三版/四版犯过)
3. 不能跨 setup 借鉴 — UDA / FedDG / personalization FL 各有约束 (三版犯过)
4. critique 要诚实 — 不能为 critique 而 critique
5. 真正的方案必须基于实测数据 + 兼容已有 framework (六版才到位)


## 发现 10: PG-DFC v2 — M1/M2 实证验证为真问题, 锁定 fix (2026-04-26 23:00)

**时间**: 2026-04-26 22:50-23:00 在 sc4 (westc:14824) 跑诊断脚本

**事件**: Codex review 给出 13 项 critique, 其中 M1 (batch-mean EMA 在小 class count 方差爆炸) / M2 (sample-weighted aggregation 不消除 domain skew) 被推断"必须改". 我们没有盲信 review, 而是写脚本 (`scripts/diagnostic/dump_client_distribution.py`) 用 F2DC 真实 dataloader 在 3 dataset × 3 seed = 9 个分布上实测.

**关键发现**: review 完全正确, 实测确认两个问题在所有 dataset 上都真实存在, 且严重程度**超过 review 预测**.

**实测数字 (3-seed mean)**:

| Dataset | M1 single_sample | M2 size_skew | 决策 |
|---|:--:|:--:|---|
| PACS | 8.5% | **4.09x** (max 6.93) | M2 必须 L2-norm 等权 |
| Office | 13.7% (max **60%**) | **7.16x** | M1+M2 都必须改 |
| Digits | **22.9%** (max 45%) | **10.17x** | M1+M2 都必须改 |

**根因**:
- M1: Office dslr client 只 25 sample → 1 个 batch 不够填,每类 1-2 样本被 EMA outlier 拽飞. Digits usps 72 / syn 74 同样问题.
- M2: PACS sketch 943 vs photo 400 (skew 7x), Office caltech 179 vs dslr 25 (skew 7x), Digits svhn 732 vs usps 72 (skew 10x). sample-weighted 让大 client 主导 prototype, 小 domain 收到偏字典反向引导.

**锁定 v2 fix**:
- **Fix 1 (M1)**: EMA → Round 内 sample 累加, round 末算 mean. 这是 FedProto/FPL 标准做法, 数值稳定无 α 调参.
- **Fix 2 (M2)**: Sample-weighted → L2-norm + 等权. 消除 magnitude 主导, 真正 domain-invariant prototype.

**未改的 (review 推断,未实测)**:
- C1/C2/C3 (proto 路径污染 mask): 先跑加诊断 hook 看 mask sparsity / r_feat ER 是否真塌缩, 数据驱动决策
- M3 (proto_weight 量级): review 数值估算偏高, 实测看现象
- M4 (attention temp): review 数学推导错 (q@k/√C 方差=1 不是 √C), 实测看现象
- m1-m4: 工程优化, 后期再说

**经验**: Review 是好的纠错工具, 但不应该全盘接受 — 必须用实测数据决定哪些 fix 必做. 实测脚本 10 分钟产出 9 个分布, 省了几周可能走错的 fix 路径.

**产出**:
- 诊断脚本: `scripts/diagnostic/dump_client_distribution.py`
- 完整 JSON: `experiments/ablation/EXP-130_F2DC_baseline_main_table/client_distribution.json` (本地已同步)
- 方案文档更新: `obsidian/2026-04-26/PG-DFC方案_xxx.md` 加附录 C (v2 实证修正) + 附录 D (改动量统计)


## 发现 11: PG-DFC v3 — Review 第二轮发现 v2 新硬伤, 锁定 4 项数学结构 fix (2026-04-26 深夜)

**时间**: 2026-04-26 23:30 review 第二轮 (NV1-NV4)

**事件**: v2 锁定 M1/M2 fix 后, review 第二轮指出 v2 引入了**新硬伤** (NV1) + 工作流分类错误 (NV2). 全部接受, 升级 v3.

**4 个新问题**:

| # | 问题 | 真假 | v2 错在哪 | v3 fix |
|---|---|:--:|---|---|
| NV1 | L2-norm 聚合 → server unit vector 跟 client raw vector magnitude 不一致 → attention 异常 | 真 ★★★★ | v2 引入新 bug | **cosine attention** (顺便解 M4) |
| NV2 | C1 (proto 反传污染 mask) 是数学必然, 不应"等实测" | 真 ★★★ | 工作流分类错 | **预先 detach proto 路径** |
| NV3 | α_round 文档歧义 (0 vs 0.5), 0.5 在数学上有 backbone 错配 | 真 ★★ | 文档歧义 | **明确 α_round=0 (不跨 round 平滑)** |
| NV4 | server 端没考虑投票成员变化, 某 client 整 round 没采到某类 → mask 后 K_c 跨 round 跳跃 → μ_global jump | 真 ★★ | 没考虑 | **server 端 EMA β=0.8** |

**Review 自己 retract M4**:
- 重新核算 dot-product attention `q@k.T/√C` 数学, 确认方差 = 1 不是 √C
- 我之前 v2 判断 "M4 数学推导错" 是对的
- 但 NV1 让 attention 即使 logits std=1 也异常 → cosine attention 一并解决

**最有价值的洞察 — 工作流分类原则**:

| 问题类型 | 决策方法 | 例子 |
|---|---|---|
| **数据驱动** | 实测验证后 fix | M1, M2 (看 client 分布) |
| **数学结构** | 看代码直接预防, 事后 ablation 验证 | C1, C2, C3, NV1, NV2 |
| **数值估算** | 实测看现象, 但要带预案 | M3, M4 |

v2 把所有未 fix 项一律"等实测"是错的. autograd 反传路径 / magnitude scale 这种"看代码就能定"的事不需要实测.

**两个 EMA 的澄清** (用户问"server EMA 又用 EMA?"):
- v1 client EMA (已取消): 跨 batch, batch_mean 噪声 + backbone 错配 (M1+NV3)
- v3 server EMA (新加): 跨 round, 输入是稳定聚合, 不涉及 backbone (NV4)
- **完全不冲突, 不是同一个东西**

**v3 锁定 5 个 fix** (含 m2 持久化):
1. cosine attention + temperature τ ∈ {0.1, 0.3, 0.5}

**v3 必跑 5 个 sanity 实验** (PACS seed=2):

- #1 full v3 → 主推荐配置

## 发现 9: F2DC release 在 PACS/Office 跑不出论文数 (2026-04-27)

**时间**: 2026-04-27 09:50 BJT (24/27 跑完, moon pacs 重跑中)

**事件**: EXP-130 完整跑完 24 个组合 (3 algo × 3 dataset × 3 seed, R=100), 提取 AVG Best mean ± std:

| Algo × Dataset | F2DC paper | 我们复现 (env) | Δ |
|---|:--:|:--:|:--:|
| F2DC × PACS | 76.47 | **63.89 ± 3.80** | **-12.6** |
| F2DC × Office | 66.82 | **61.19 ± 2.57** | -5.6 |
| F2DC × Digits | 87.23 | **92.28 ± 1.73** | +5.0 |

| FedAvg × Digits | 81.24 | 90.15 ± 1.29 | +8.9 |
| MOON × Digits | 76.60 | 89.23 ± 2.76 | +12.6 |

**关键观察**:
1. **F2DC 在 PACS 上比 FedAvg 只高 +2pp** (63.89 vs 61.88), 不是论文宣称的 +10pp
2. **PACS 掉 12.6pp, Office 掉 5.6pp** — release 代码 (含我们 8 个 patch) 跑不出论文 76.47/66.82
3. **Digits 三方法都比论文高 5-12pp** — 可能是我们 protocol 跟论文不完全一致 (digits 1% 抽样太敏感) 或 release 数据 partition 实际比论文更友好
4. **F2DC release 缺论文 4.3 节 Domain-Aware Aggregation 实现** + DFD/DFC private 与论文文字不一致 — 这就是差距来源

**含义 (paper 角度)**:
- ✅ **paper 主表用我们环境复现的数字做 baseline**, 不用论文报数 (公平比较)
- ✅ **FedDSA 主表数字直接超**: PACS orth_only 80.64% 领先 F2DC 复现 63.89% **达 16.7pp**, Office 89.09% 领先 F2DC 61.19% **达 28pp**
- ✅ **写 paper 时加 footnote**: "We reproduced F2DC using its official release + 8 necessary patches and obtained the numbers reported here. The paper-reported 76.47% on PACS could not be reproduced from the release code (likely missing the Domain-Aware Aggregation module described in Sec 4.3)."
