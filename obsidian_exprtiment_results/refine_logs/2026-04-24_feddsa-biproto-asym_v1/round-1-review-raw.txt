**Date**: 2026-04-24
**Stage**: Round-0 Initial Proposal
**Target**: 在 FDSE_CVPR25 框架下, 3-seed mean AVG Best 严格胜过 FDSE 本地复现 baseline (PACS ≥ 80.91, Office-Caltech10 ≥ 91.08), 同时交付 t-SNE / probe ladder / feature metrics 三套可视化 evidence.

## Technical Gap

当前四派方法的失败点:
1. "擦除派" (FDSE/FedDP/FedSeProto): domain 当噪声擦除, 不对 domain 建模
2. "私有派" (FedSTAR/FedBN): 风格本地, domain 被降级为 nuisance
3. "共享不解耦派" (FISC/StyleDDG/FedCCRL): 混合空间 AdaIN 导致负迁移
4. "feature-level 对抗派" (CDANN): EXP-108 证伪, probe 0.96 无 accuracy 增益

缺失的机制: 目前所有 FL cross-domain 解耦工作对 "domain 信息" 的处置只有三种: 擦除 / 私有 / 本地对抗. 从未有工作**把 domain 升级为一阶 federated 对象** —— 即像 class prototype 一样在 server 端维护跨 client 的 domain prototype bank, 并在 prototype 级 (而非 feature 级) 强制与 class prototype 几何互斥.

同时, 目前 FL 里**独立 style encoder 几乎都是对称双 trunk** (2× AlexNet), 忽视了 "style ≈ (μ, σ) 统计"的强先验. 一个**轻量非对称统计 encoder** 可以在 <2% 参数膨胀下提供 proper inductive bias.

## Method Thesis

**One-sentence thesis**: 把 domain 从"要擦除的噪声"升级为"联邦共享的一阶原型 (Pd)", 与 class prototype (Pc) 通过 proto-level 几何互斥并列存在; 用非对称统计 encoder 给 Pd 提供 inductive bias; 整套机制在 loss 数目和参数量上保持极简.

**Why smallest adequate**: 相较 CDANN 需 GRL+dom_head+对抗 (已证伪), BiProto 只在 orth_only 基础上加 1 个轻量 encoder (~1M) + 2 条 proto-level InfoNCE + 1 条 proto-cos 排斥. 所有 loss 都在 proto 空间 (低维 K×d, K≤10), 梯度冲突半径极小, 避开 EXP-017/095/108 的全部已知坑.

## Contribution Focus

- **Dominant (C1)**: Federated Domain Prototype (Pd) as a first-class shared object, orthogonal-to-class at the prototype level via cosine exclusion. 首次在 FL cross-domain 将 domain 作为与 class 对称的一阶联邦对象建模. 机制: Pd 由非对称统计 encoder 生成, 跨 client 直接 concat (无聚合歧义, 因为每 client 一个 domain). 监督: Pd 通过 sty-proto InfoNCE 驱动 encoder_sty 学习 same-domain attraction / cross-domain repulsion, 同时 Pd ⊥ Pc 的 proto-level cosine 损失在低维空间强制互斥.
- **Supporting (C2)**: Asymmetric statistic encoder — 风格 encoder 只做 (μ,σ) 统计提取 + 轻量 MLP (~1M, <2% of 60M AlexNet). 物理上无法编码 class-level 判别信息 (丢弃空间结构), 从架构切断 "sty 承载 class" 泄漏, 无需对抗 loss.
- **Explicit non-contributions**: 不 claim MI-optimal / 不 claim 通信效率 / 不做收敛证明 / 不跨 backbone / 不做 DP privacy.

### Core Mechanism: Pd

## Core Principles

### 1. The Review Mindset

**Goals of Code Review:**

- Catch bugs and edge cases
- Ensure code maintainability
- Share knowledge across team

**Not the Goals:**

### 2. Effective Feedback

**Good Feedback is:**

- Specific and actionable
- Educational, not judgmental
- Focused on the code, not the person
- Balanced (praise good work too)
- Prioritized (critical vs nice-to-have)


**What Not to Review Manually:**
