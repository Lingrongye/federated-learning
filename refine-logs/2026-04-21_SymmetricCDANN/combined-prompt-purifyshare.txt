## 1. 数学直觉 (pipeline 拆解)

### 1.1 当前症状 (为什么需要 3-stage)

双头解耦 `cos⊥ + HSIC` 只是**统计独立约束**, 无方向性:

```
encoder(x) → feat(1024d, 包含 class+domain+ 其他信息)
           ├→ sem_head(128×1024 Linear) → z_sem
           └→ sty_head(128×1024 Linear) → z_sty

约束: z_sem 和 z_sty 方向正交 + 低 HSIC
```

数学上, **正交不等于分工**:
- `span(z_sem) ⊥ span(z_sty)` 只是两子空间正交
- **class 信号在整个 1024d feature 上, 在两个子空间里都有投影**
- 所以 probe_sem_class ≈ probe_sty_class ≈ 96% (实测 Office)

**核心缺失**: 没有约束说 **"class 信号**只**落在 span(z_sem), 不落在 span(z_sty)"**.

### 1.2 Stage 1 — Purify (真解耦)

**目标**: 让 z_sty 里 class 可预测度掉到 random level (~1/K), 且 z_sem 里 domain 可预测度掉到 1/N.

#### 选项 A: 对抗式 (GRL-based)

```math
L_cls_sty_adv = -λ_a · CE(y, cls_head_on_sty(GRL(z_sty, λ_a)))  # 反向梯度
L_dom_sem_adv = -λ_a · CE(d, dom_head_on_sem(GRL(z_sem, λ_a)))

L_purify = L_orth + λ_h · L_hsic + L_cls_sty_adv + L_dom_sem_adv
```

**风险** (Codex 已指出):
- shared-trunk contamination: L_task 逼 encoder 在 1024d trunk 里全局含 class, GRL 只 push 一个投影藏起
- 可能 class 信号往 **nonlinear manifold** 迁, linear probe 查不到, 但 MLP probe 还能查出
- BN leakage: FedBN 的 client-specific BN stats 让 domain 永远可分

#### 选项 B: Mutual Information Bottleneck

```math
L_mi_sty_class = βI · I(z_sty; y)   # upper bound via MINE/CLUB
L_mi_sem_domain = βI · I(z_sem; d)
```

CLUB (Upper bound on MI) 比 adversarial 更稳, 直接约束 MI.

**风险**:
- MINE/CLUB 在 small batch FL 里训不稳 (EXP-078 NaN 先例)
- 需要额外 critic network, 增加参数
- PACS 每 client ~300 样本, MI estimator 样本不足

#### 选项 C: Reconstruction (AE 式)

```math
z_combined = concat(z_sem, z_sty)
x_recon = decoder(z_combined)
L_recon = ||x - x_recon||²

# 同时保持两边不能单独重建:
# z_sem alone 无法重建 domain-specific 外观
# z_sty alone 无法重建 class-specific 几何
```

SCFlow (ICCV 2025) 的思想. 要可逆 + 双头信息互补.

**风险**:
- Decoder 训练重 (AlexNet encoder 对应 decoder 几十 M 参数)
- Flow matching 太复杂, 简化版可能退化

#### 选项 D (推荐): A + B 组合

adversarial 作主力 + MI 约束作辅助 + **nonlinear probe** 作验证:

```math
L_purify = L_orth + λ_h · HSIC + 
           λ_a · [adversarial class → z_sty, adversarial domain → z_sem] +
           λ_m · [MI(z_sty; y) upper bound + MI(z_sem; d) upper bound]

验证: MLP probe (2-layer) 在训完后测:
- probe_sty_class (MLP) ≤ 30% → 真解耦
- probe_sem_class (MLP) ≥ 80%
- probe_sty_domain (MLP) ≥ 90%
- probe_sem_domain (MLP) ≤ 40% (还是难, 因 FedBN)
```

### 1.3 Stage 2 — Prototype Alignment on 纯净 z_sem

**目标**: 在 z_sem 已纯 (只含 class) 前提下, 用 InfoNCE 拉近同类跨 client 的 z_sem, 推远不同类.

#### 公式

```math
# Global class prototype (server-maintained, EMA 更新):
p_c ← α · p_c + (1-α) · mean_k(z_sem_c_client_k_mean)   # c: class, k: client

# Client-side InfoNCE:
L_infonce = -Σ_i log [ exp(cos(z_sem_i, p_{y_i}) / τ) / Σ_c' exp(cos(z_sem_i, p_{c'}) / τ) ]
```

**Drift Penalty**: 防止 client-local z_sem 被 InfoNCE 拉到和其他 client 全一样 (会失去 client-specific 辨识力, 且聚合后 BN 漂移会崩):

```math
L_drift = Σ_c ||z_sem_c_local_mean - p_c_EMA||²  # proximal 约束

或: ||z_sem_k - z_sem_k_{t-1}|| < δ  # trust-region 式更新
```

EMA 的 α 应该偏大 (0.95+) 让 p_c 缓慢更新, 避免震荡. 这是 FedProto/FPL 的典型套路.

#### 为什么这次可能 work (之前 EXP-076 bell 调度失败的原因)

- EXP-076: InfoNCE 在没有真解耦的 z_sem 上训 → z_sem 里已有 domain 信号 → 跨 client 拉近 z_sem 把 domain 信号也对齐了 → 破坏 domain-specific 识别能力 → 精度掉
- 现在 Stage 1 真解耦 → z_sem 理论上只有 class → 跨 client 对齐 class 表示**对的事**
- Drift penalty 避免 InfoNCE 过度拉齐, 保持 client-local 细节

### 1.4 Stage 3 — Style Share on 纯净 z_sty

**目标**: 用各 client 共享的纯 z_sty 做数据增强, 扩 z_sem 的泛化能力.

#### 公式 (feature-space AdaIN)

```math
# Server 收集并广播风格仓库:
S_global = {(μ_k, σ_k) for k in clients}  # z_sty 每维的 mean/std

# Client-side augmentation (per-batch):
for sample (x, y):
    z_sem = sem_head(encoder(x))
    z_sty = sty_head(encoder(x))
    
    # 采样 external style:
    (μ_ext, σ_ext) ~ S_global \ {local}
    
    # 注入到 z_sem (channel-wise AdaIN):
    z_sem_norm = (z_sem - μ_local) / (σ_local + ε)
    z_sem_aug = σ_ext * z_sem_norm + μ_ext  
    
    # 分类头:
    logit_aug = sem_classifier(z_sem_aug)
    logit = sem_classifier(z_sem)
    
    L_task = CE(y, logit) + λ_aug · CE(y, logit_aug)
```

**关键**: AdaIN 统计量来自 z_sty (纯风格), 注入到 z_sem (纯类). **这次不会污染 class**.

之前 EXP-059/067 失败 = z_sty 不纯 → AdaIN 把 class 信号传到 z_sem 导致 z_sem 被污染.

### 1.5 Total Loss (联合训练)

```math
L = L_task 
  + λ_orth · L_orth + λ_hsic · HSIC
  + λ_adv · (L_cls_sty_adv + L_dom_sem_adv)    # Stage 1
  + λ_infonce · L_infonce(z_sem, p_c)           # Stage 2
  + λ_drift · L_drift                           # Stage 2 约束
  + λ_aug · L_task_aug                          # Stage 3 风格共享增强
  + [optional MI terms]
```

超参 estimate (保守起点):
- λ_orth = 1.0 (已有)
- λ_hsic = 0.1 (已有)
- λ_adv = 0.5 (比 CDANN 的 1.0 保守, 避免训崩)
- λ_infonce = 1.0, τ = 0.1
- λ_drift = 0.01 (很小, 只作软约束)
- λ_aug = 0.3 (不要压过主 CE)

## 3. Risk Assessment (诚实)

### 3.1 最大风险: Stage 1 可能根本达不到

Codex R1 review 明确说:
> "Both branches are cheap linear views of the same shared encoder state. Another GRL can reduce one easy readout, but it does not give you identifiability."

**意思**: 双头 Linear 投影 + shared encoder, 无论怎么加约束可能都压不住 class 泄漏. 因为:
- encoder trunk 已含 class (CE loss 逼的)
- linear 投影从 trunk 出 → class 信号在投影里普遍存在
- 要真解耦, 必须把 class 信号从 trunk 里挤压到特定子空间 — 这需要**重新训练整个 encoder**, 不只是头

**缓解**: 
- 把 sem_head/sty_head 从 Linear 升到 2-3 层 MLP (增加 nonlinear 分工能力)
- encoder 的 L_task 梯度只经 sem_head, **sty_head 端截断 encoder 回流** (让 sty_head 自己学, 不影响 encoder)

### 3.2 Stage 2 InfoNCE 的两难

**FedProto / FPL / FedPLVM 这些工作的 InfoNCE 用 class 原型**, 但它们**都没做真解耦**, 所以它们的 z 本身就混合. 我们真解耦之后的 z_sem, InfoNCE 作用模式**不一样** (更干净但也更敏感).

**可能**: 纯 z_sem 对 InfoNCE 过度敏感 → 训不稳 → 需要 drift penalty 调得很精细

### 3.3 Stage 3 AdaIN 未必比 Plan A 好

即使 z_sty 真纯, feature-space AdaIN 不一定比不做增强好. 因为:
- AlexNet from scratch 小网络容易过拟合于训练 distribution
- AdaIN 注入 external style = 制造假样本 → encoder 可能"不认识"这些假样本的类
- 需要 λ_aug schedule 小心调

L_align = λ_a · Σ -cos(z_sem, p_{y}) / τ                  # alignment

# DCFL 建议 λ_a = 0.9, λ_u = 0.1

L_drift_OT = OT_assignment(z_sem_c_local, p_c_EMA)        # FedAli (arXiv 2411.10595)
